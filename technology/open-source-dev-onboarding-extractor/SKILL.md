# 开源项目开发者 Onboarding 信息提取

从任意开源仓库提取**面向开发者**的项目情报。产出应帮助软件工程师建立足够的心智模型，以**定位代码、追踪主链路、本地运行、安全提交第一次改动**。

本 Skill **不**用于终端用户营销。不要优化「说服安装」；应优化**理解、导航、验证**。

## 前置条件

- 对目标仓库有读权限（本地 clone 或工作区）
- 在安全可行时可用 Shell 执行构建/测试命令
- 有写权限保存草稿与最终报告

## 操作原则

1. **证据优先** — 每条结论须引用文件路径、配置键、文档章节或命令输出。未知项标记为 `未验证` 或 `未找到`。
2. **一条主链路** — 在展开广度之前，先确定最具代表性的一条端到端路径。
3. **按能力建图，不按目录扫街** — 按系统*做什么*组织，而非按文件夹字母序。
4. **公开 vs 内部** — 区分贡献者文档、公开 API/SDK 与内部模块。
5. **先读后写** — 完成侦察后再撰写最终报告。
6. **深度与体量匹配** — 小型 CLI 工具可少做容器分析；平台型项目需覆盖全部十个方面。
7. **中文产出** — 草稿与最终报告（含检查清单、议程、Mermaid 图内节点标签与说明）**一律使用简体中文撰写**。用户明确要求其他语言时除外。代码路径、命令、配置键、符号名、仓库内专有标识保留原文。

## 工作流

复制以下清单并跟踪进度：

```
提取进度：
- [ ] 阶段 0：侦察
- [ ] 方面 1：问题域与边界
- [ ] 方面 2：系统上下文
- [ ] 方面 3：主执行链路
- [ ] 方面 4：架构原则
- [ ] 方面 5：代码地图
- [ ] 方面 6：配置与状态
- [ ] 方面 7：构建、运行、调试
- [ ] 方面 8：测试与质量门禁
- [ ] 方面 9：扩展与贡献
- [ ] 方面 10：演进与地雷
- [ ] 阶段 3：合成报告 + 必读 10 文件 + 检查清单
```

### 阶段 0：侦察（15～30 分钟）

1. 按序阅读：`README*`、`CONTRIBUTING*`、`ARCHITECTURE*`、`docs/` 索引、`CHANGELOG*`、包清单（`package.json`、`go.mod`、`Cargo.toml`、`pyproject.toml` 等）。
2. 识别：语言、monorepo 或单包、入口（`cmd/`、`main.*`、`src/index.*`）、CI 配置（`.github/workflows` 等）。
3. 草稿保存至 `dev-onboarding-scratch.md`（仓库根目录或用户指定输出目录）。
4. 当你能用**一段技术文字**描述项目时停止。

各方面的发现提示与典型文件位置见 [references/extraction-checklist.md](references/extraction-checklist.md)。

### 阶段 1：提取十个方面

对每个方面，用 checklist 收集证据，写入草稿后再进入下一项。**不要跳过** — 若确实无关（如无插件模型），写 `不适用：<原因>`。

| # | 方面 | 最低交付物 |
|---|------|------------|
| 1 | 问题域与边界 | 问题陈述、角色、范围内/外、成功指标 |
| 2 | 系统上下文 | 上下文图 + 容器、外部依赖、部署单元 |
| 3 | 主执行链路 | ≥6 步，带文件/函数锚点 |
| 4 | 架构原则 | 文档化规则 + 推断约束、依赖方向 |
| 5 | 代码地图 | 能力表：域 → 入口 → 测试 |
| 6 | 配置与状态 | 配置来源、优先级、密钥、持久化 |
| 7 | 构建、运行、调试 | 精确命令、环境要求、日志、常见失败 |
| 8 | 测试与质量门禁 | 测试分层、CI 检查、PR 前命令 |
| 9 | 扩展与贡献 | 插件/API 模型、PR 期望、CODEOWNERS |
| 10 | 演进与地雷 | 遗留路径、重构、禁区、文档/代码漂移 |

**方面 3 是主干。** 若时间受限，优先完成 1～3 和 7～8，再完善 9～10。

### 阶段 2：现场验证（可行时）

跑最薄的证明路径：

1. 文档中的安装/构建命令（或 `未执行：<原因>`）。
2. 与主链路相关的一条窄测试或 smoke 命令。
3. 在方面 7/8 中记录精确命令与结果。

除非用户要求，不要跑完整 CI。优先单文件或单包测试。

### 阶段 3：合成最终报告

1. 复制 [templates/developer-onboarding-report.md](templates/developer-onboarding-report.md)。
2. 填满各节；用仓库具体证据替换占位符。
3. 附加：
   - **必读 10 文件**（有序，每文件一行理由）
   - **开发者理解检查清单**（能否画上下文、追踪主链路、本地运行？）
   - **建议 onboarding 议程**（30 分钟 / 2 小时版）
4. 至少包含 **两张 Mermaid 图**：系统上下文或容器图 + 主链路时序图（图中节点标签与说明用中文，技术组件名可保留英文）。
5. 交付报告。默认文件名 `developer-onboarding-report.md`，除非用户指定路径。
6. **全文中文**：章节标题、叙述、表格说明、检查清单、议程均为简体中文；路径、命令、函数名、配置键保留原文。

## 产出质量门槛

交付前核对：

| 检查项 | 通过标准 |
|--------|----------|
| **中文** | 产出文档主体为简体中文（用户指定其他语言时除外） |
| 有据 | 不编造命令、端口或模块名 |
| 可导航 | 主链路 ≥6 步，含仓库根相对路径 |
| 可执行 | 方面 7 有可复制粘贴的运行命令 |
| 诚实 | 未知项与文档/代码冲突明确标注 |
| 边界清晰 | 公开 API 与内部分离 |
| 图示 | ≥2 张 Mermaid 图，语法正确 |

## 深度模式

| 模式 | 适用 | 范围 |
|------|------|------|
| **快速** | 用户限时 ≤30 分钟 | 方面 1～3、5（精简）、7；约 5 页报告 |
| **标准** | 默认 | 全部十个方面；完整报告 |
| **深度** | Monorepo / 平台 | 标准 + 主要容器的分包子报告 |

根据用户请求询问或推断模式。默认 **标准**。

## 反模式

- 不要产出用户营销文案（痛点、安装 CTA），除非用户单独要求。
- 不要罗列每个目录 — 代码地图能力行上限约 12 行。
- 不要把原始 `tree` 输出 dump 进报告。
- 未实际运行时不要声称测试/构建成功；应标记 `未验证`。
