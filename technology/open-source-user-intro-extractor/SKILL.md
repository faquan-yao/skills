# 开源项目使用者向信息提取

从任意开源仓库提取**面向软件使用者（非开发者）**的项目信息，按**六段叙事**组织为一份结构化文档。

## 路径约定

**{项目根}** = 被分析开源仓库的根目录（通常为工作区根）。阶段 0 须确认。

**{产出目录}** = 用户指定的输出目录；若用户未指定，默认为 `{项目根}/项目汇总/`。阶段 0 须确认并创建（不存在则 `mkdir`）。

## 产出路径（必遵）

| 文件 | 路径 |
|------|------|
| 草稿 | `{产出目录}/user-intro-scratch.md` |
| 最终文档 | `{产出目录}/user-project-info.md` |

用户指定输出目录时，所有产物直接写入该目录下，不再追加 `项目汇总/` 子目录。

## 前置条件

- 对目标仓库或官方文档有读权限
- 可选：访问项目官网、docs 站点
- 有写权限保存草稿与最终文档

## 操作原则

1. **先读后写** — 结论须有据可查（对照 README、docs、官网）；未知标 `未验证` 或 `未找到`。依据记入草稿，不写入最终报告。
2. **用户语言** — 写「在 Telegram 里聊天」，不写「Channel Plugin 入站」。
3. **诚实门槛** — 写明安装难度、API Key、命令行、自托管等真实前提。
4. **开源双面** — 写清值得自托管的理由，以及不适合只想注册即用的人。
5. **信息而非演讲** — 输出是**可引用的项目事实与定位**，不是说服性脚本。
6. **中文产出** — 全文简体中文。用户指定其他语言时除外。命令、URL、路径、专有名词保留原文。

## 工作流

```
提取进度：
- [ ] 阶段 0：使用者向侦察
- [ ] 第 1 段：钩子
- [ ] 第 2 段：痛点
- [ ] 第 3 段：转变
- [ ] 第 4 段：差异化
- [ ] 第 5 段：降低行动成本
- [ ] 第 6 段：信任收尾
- [ ] 阶段 2：合成文档
```

### 阶段 0：使用者向侦察（10～20 分钟）

按序阅读：

1. `README*` 开头与 Quick Start
2. `docs/start*`、`docs/getting-started*`、安装/向导文档
3. 官网或 docs 首页
4. `docs/help/faq*`
5. 功能/渠道/平台概述（channels、features、platforms）
6. **不深入** `src/`、架构 internals、CONTRIBUTING（除非影响用户门槛）

草稿：`{产出目录}/user-intro-scratch.md`。  
停止条件：能写出**一句用户向定位** + **一个具体使用场景**。

提取问题见 [references/six-segment-guide.md](references/six-segment-guide.md)。

### 阶段 1：提取六段信息

每段写完再进下一段。不可跳过；无关则写 `不适用：<原因>`。

| 段 | 名称 | 提取内容 |
|----|------|----------|
| 1 | 钩子 | 一句话定位 + 具体使用画面 |
| 2 | 痛点 | 2～3 条场景化痛点及本产品回应 |
| 3 | 转变 | Before / After / Because；happy path；首次成功标志 |
| 4 | 差异化 | vs 替代方案、开源卖点（精简） |
| 5 | 降低行动成本 | ≤3 步上手、耗时、前置条件、验证方式、排障入口 |
| 6 | 信任收尾 | 适合/不适合谁、社会证明（若有）、主 CTA、文档/社区链接 |

时间受限时优先 1、2、3、5、6。

### 阶段 2：合成文档

1. 按 [templates/user-project-info.md](templates/user-project-info.md) 结构输出。
2. 填满六段；可选补充「日常能力摘要」「安全与隐私（用户向）」。
3. 交付至 `{产出目录}/user-project-info.md`。

### 可选：轻量核对

仅核对 README 与 docs 中安装步骤是否一致；**不要**为此 Skill 跑完整构建/测试。

## 产出质量门槛

| 检查项 | 通过标准 |
|--------|----------|
| **中文** | 文档主体为简体中文 |
| 用户可读 | 非开发者能读懂 |
| 有画面 | 第 1 段有具体场景 |
| 诚实 | 门槛与「不适合谁」已写明 |
| 可查证 | 关键声称与 README/docs 一致，无编造 |
| 有据 | 不编造命令、功能或链接 |

## 深度模式

| 模式 | 适用 | 范围 |
|------|------|------|
| **快速** | 时间紧 | 六段各 2～4 句，省略可选节 |
| **标准** | 默认 | 完整六段 + 能力摘要 |

默认 **标准**。

## 反模式

- 不要默认生成「读完应能回答」类检查清单。
- 不要写开发者术语堆砌（Gateway、monorepo 等），除非 docs 面向高级用户。
- 不要功能清单超过 7 条。
- 不要隐瞒 API Key、命令行、自托管门槛。
