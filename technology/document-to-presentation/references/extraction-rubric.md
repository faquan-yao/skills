# 关键信息提取规则（Fact Kernel）

从源文档抽取「事实内核」时使用本规则。只写**可验证陈述**；无法从文档证实则标 `文档未写明`。

## 六类必抽信息

| 类型 | 定义 | 通过标准 |
|------|------|----------|
| 主结论 | 听众 30 秒内应带走的唯一主张 | 一句话，含「谁 + 做什么 + 价值」 |
| 支柱论点 | 支撑主结论的 3～5 个独立论点 | 每条有主张 + 证据章节/数据 |
| 边界 | 做什么 / 不做什么 | 与文档「范围外」「不适合谁」一致 |
| 主流程 | 核心 happy path 或架构 | 3～5 步；技术文档可保留精简 mermaid |
| 行动项 | 听众下一步 | 可执行命令、链接或决策 |
| 附录素材 | 细节表、依赖、FAQ | 不进主链路，备 Q&A |

## 优先级（时间紧时）

1. 主结论 + 边界（避免误导）
2. 支柱论点 1～3
3. 主流程 + 行动项
4. 其余支柱 + 附录

## 删减原则

| 源文档规模 | 主链路页数 | 附录页数 |
|------------|------------|----------|
| 使用者向 6 段叙事（< 200 行） | 10～14 | 1～3 |
| 开发者 onboarding 十章（400+ 行） | 12～18 | 2～4 |
| README / PRD（中等） | 8～12 | 1～2 |

**删减顺序（先删）：** 实现细节、完整依赖表、次要角色、历史背景、重复表格。

**保留顺序（后删）：** 主结论、边界、核心用例、差异化、CTA。

## 证据标注

在 scratch 中为每条主张标注来源，格式：

```
主张：……
来源：文档 §3.2 / 第 47 行 / 表格「痛点」
```

最终 `fact-kernel.md` 可省略行号，但须能回溯 scratch。

## 文档类型压缩策略

### 六段使用者叙事（钩子→信任）

| 段 | 幻灯片映射 |
|----|------------|
| 钩子 | 封面 + 场景页 |
| 痛点 | 1～2 页（表格精简为 3 条） |
| 转变 | Before/After 1 页 |
| 差异化 | 1 页对比 |
| 降低行动成本 | 上手步骤 1 页 |
| 信任收尾 | 适合谁 + CTA 1 页 |

### 十章开发者 onboarding

| 章 | 幻灯片映射 |
|----|------------|
| 问题域 | 背景 + 边界 |
| 系统上下文 | 架构图 1 页 |
| 主执行链路 | 流程 1～2 页 |
| 架构原则 | 原则列表 1 页 |
| 代码地图 / 配置 | 附录或 1 页速查 |
| 构建测试贡献 | 贡献者 CTA |
| 演进与地雷 | 风险页 |

### 通用 Markdown

1. 读 H1/H2 建目录
2. 摘要/TL;DR → 主结论
3. 表格 → 支柱或附录
4. 代码块 → 仅保留命令与 3～8 行示例
5. mermaid → 精简节点后上屏

## 反模式

- 不要把「有趣但次要」的故事当支柱论点
- 不要合并两个独立主张为一页（除非时长极短）
- 不要从口播灵感反推事实（顺序永远是：文档 → kernel → PPT/口播）
